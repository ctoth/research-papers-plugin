# Extract Claims: $ARGUMENTS

Extract propositional claims from a research paper and produce or enrich a `claims.yaml` file conforming to the propstore claim schema.

## Step 0: Parse Arguments and Determine Mode

```bash
paper_dir=""
mode="enrich"

for arg in $ARGUMENTS; do
  if [[ "$arg" == "--mode" ]]; then
    continue
  elif [[ "$arg" == "enrich" || "$arg" == "create" ]]; then
    mode="$arg"
  else
    paper_dir="$arg"
  fi
done
```

Validate the paper directory:

```bash
ls "$paper_dir"/notes.md 2>/dev/null
ls "$paper_dir"/claims.yaml 2>/dev/null
```

**Decision tree:**
- `notes.md` missing -> STOP. Report: "Cannot extract claims without notes.md. Run paper-reader first."
- `claims.yaml` exists AND mode is `enrich` -> proceed to **Mode 1: Enrich**
- `claims.yaml` missing AND mode is `enrich` -> switch to create mode, report: "No claims.yaml found. Switching to create mode."
- mode is `create` -> proceed to **Mode 2: Full Creation**

---

## Mode 1: Enrich Existing Claims

Read the existing `claims.yaml` and the paper's `notes.md`. Improve each claim with real data from the paper.

### Step 1.1: Read Source Material

Read these files:
- `<paper_dir>/claims.yaml` — the mechanical extraction to enrich
- `<paper_dir>/notes.md` — the full paper notes with equations, parameters, context

If available, also read:
- `<paper_dir>/paper.pdf` or page images in `<paper_dir>/pngs/` — for verifying page numbers
- Concept registry files in the propstore (`concepts/*.yaml`) — for resolving concept IDs

### Step 1.2: Enrich Each Claim

Walk through every claim in the file. For each one, improve these fields:

#### Page Numbers (provenance.page)
Replace `page: 0` with the actual page number where the claim appears in the paper. Cross-reference:
- The parameter name against sections in notes.md
- Table/figure references
- Section headings mentioned in notes.md "Figures of Interest" or equation locations

If the exact page cannot be determined, add `provenance.section` with the section name instead and leave page as 0.

#### Concept Resolution (concept)
The generator uses `_concept_name_from_param()` which produces lowercase underscore names like `fundamental_frequency`. Check these against the concept registry:

```bash
ls concepts/*.yaml 2>/dev/null
```

For each concept name in a claim:
- If a matching concept file exists (e.g., `concepts/fundamental_frequency.yaml`), read it and use its `id` field (e.g., `concept1`)
- If no match, check concept aliases — read each concept file and compare against the `aliases` list
- If still no match, keep the descriptive name. Do NOT invent concept IDs.

#### SymPy Expressions (sympy)
For equation claims, the generator copies the LaTeX expression into both `expression` and `sympy`. Replace the `sympy` field with a valid SymPy-parseable expression:
- LaTeX `\frac{a}{b}` -> SymPy `a/b`
- LaTeX `e^{x}` -> SymPy `exp(x)`
- LaTeX `\sqrt{x}` -> SymPy `sqrt(x)`
- LaTeX `\sum_{i=0}^{N}` -> SymPy `Sum(f(i), (i, 0, N))`
- Subscripts like `T_p` -> SymPy `T_p` (use underscores)

Verify the sympy string is parseable: it should use standard SymPy function names and operators.

#### Variable Bindings (variables)
For equation claims, populate the `variables` list with mappings from equation symbols to concepts:

```yaml
variables:
  - symbol: "F0"
    concept: "concept1"
    role: "dependent"
  - symbol: "Ps"
    concept: "concept2"
    role: "independent"
```

Roles: `dependent`, `independent`, `parameter`, `constant`.

#### Conditions (conditions)
Add CEL expressions where the paper specifies conditions under which a claim holds:
- Population: `"sex == 'male'"`, `"sex == 'female'"`
- Task: `"task == 'speech'"`, `"task == 'singing'"`
- Context: `"vowel == 'a'"`, `"register == 'modal'"`
- Methodology: `"measurement_method == 'inverse_filtering'"`

Use consistent condition vocabulary across claims in the same file. Conditions are equality checks by default; use comparison operators for ranges.

#### Notes (notes)
Add a `notes` field when methodological context is important:
- Sample size or population details not captured in conditions
- Measurement methodology caveats
- Known limitations the authors acknowledge
- Whether a value is measured vs. derived vs. assumed

#### Stances (stances)
Add stances for cross-paper relationships when evident from the paper's discussion:

```yaml
stances:
  - type: supported_by
    target: "claim3"  # claim ID in another paper's claims.yaml
    strength: "strong"
    note: "Replicates finding with larger sample"
  - type: contradicted_by
    target: "claim7"
    strength: "moderate"
    conditions_differ: "Different vowel context"
```

Stance types: `supported_by`, `contradicted_by`, `superseded_by`, `mechanism_for`.

**Only add stances you can justify from the paper text.** Do not speculate about relationships.

#### Uncertainty (uncertainty, uncertainty_type)
If the paper reports uncertainty for a parameter value, add:

```yaml
uncertainty: 0.29
uncertainty_type: sd  # sd, se, ci95, or range
```

#### Sample Size (sample_size)
If the paper reports how many observations a parameter estimate is based on, add `sample_size`.

#### Measurement Claims
If the paper contains perceptual or behavioral measurements (JND thresholds, preference ratings, detection thresholds), ensure these are typed as `measurement` with:
- `target_concept`: the concept being measured
- `measure`: one of `jnd_absolute`, `jnd_relative`, `discrimination_threshold`, `preference_rating`, `detection_threshold`, `correlation`, `effect_size`
- `value` and `unit`: the measurement result
- `listener_population`: if specified (e.g., `"native_english"`, `"trained_musicians"`)
- `methodology`: brief description of experimental method

### Step 1.3: Add Missing Claims

After enriching existing claims, check if the notes contain information not captured:
- Parameters mentioned in prose but not in tables
- Observations in the Discussion or Results not extracted as testable properties
- Model descriptions that should be `model` type claims
- Key findings stated as conclusions

Add new claims with sequential IDs continuing from the highest existing ID.

### Step 1.4: Write Output

Write the enriched claims back to `<paper_dir>/claims.yaml`. Preserve YAML formatting:

```yaml
source:
  paper: Author_Year_ShortTitle
  extraction_model: "claude-opus-4-20250514"  # or whatever model you are
  extraction_date: "2026-03-17"  # today's date

claims:
  - id: claim1
    type: parameter
    concept: concept1
    value: 200.0
    unit: Hz
    conditions:
      - "task == 'speech'"
    provenance:
      paper: Author_Year_ShortTitle
      page: 5
      table: "Table 1"
      section: "Results"
      quote_fragment: "Mean F0 was 200 Hz for speech tasks"
    notes: "Measured via electroglottography, N=24 male speakers"
```

## Mode 2: Full Creation

Create `claims.yaml` from scratch using `notes.md` content. Use this when `generate_claims.py` cannot extract structured data (unusual formatting, no parameter tables, etc.).

### Step 2.1: Read Source Material

Read:
- `<paper_dir>/notes.md` — primary source
- `<paper_dir>/paper.pdf` or page images — for page numbers and verification
- Concept registry (`concepts/*.yaml`) — for concept ID resolution

### Step 2.2: Extract Parameter Claims

For each parameter, constant, or threshold mentioned in the paper:

```yaml
- id: claim1
  type: parameter
  concept: <concept_id or descriptive_name>
  value: <number>
  unit: <unit string>
  conditions:
    - "<CEL expression>"
  provenance:
    paper: <paper_dir_name>
    page: <page number>
    section: "<section name>"
    quote_fragment: "<brief supporting quote>"
  notes: "<methodological context>"
```

Rules:
- Every parameter needs at minimum: id, type, concept, provenance
- Include `value` OR `lower_bound`+`upper_bound` (at least one)
- Always include `unit` when known
- Use concept IDs from the registry when available; otherwise use lowercase_underscore names

### Step 2.3: Extract Equation Claims

For each equation or mathematical relationship:

```yaml
- id: claim5
  type: equation
  expression: "F_0 = k \\cdot P_s^{0.5}"  # human-readable
  sympy: "F_0 - k * P_s**0.5"  # SymPy-parseable (rearranged to equal zero, or as assignment)
  variables:
    - symbol: "F_0"
      concept: "concept1"
      role: "dependent"
    - symbol: "P_s"
      concept: "concept2"
      role: "independent"
    - symbol: "k"
      concept: "proportionality_constant"
      role: "parameter"
  provenance:
    paper: <paper_dir_name>
    page: <page number>
    section: "<section>"
```

Rules:
- `expression`: human-readable LaTeX-style string
- `sympy`: valid SymPy expression using standard functions (exp, sqrt, log, Sum, etc.)
- Every symbol in the equation must have a variable binding
- Include `fit` statistics (r, r_sd, slope, slope_sd) when the paper reports them

### Step 2.4: Extract Observation Claims

For qualitative claims, empirical observations, and testable properties:

```yaml
- id: claim8
  type: observation
  statement: "Fundamental frequency increases with subglottal pressure in a roughly logarithmic relationship."
  concepts:
    - concept1
    - concept2
  conditions:
    - "register == 'modal'"
  provenance:
    paper: <paper_dir_name>
    page: <page number>
    section: "Discussion"
    quote_fragment: "Our results show a logarithmic dependency..."
```

Rules:
- `statement`: a single declarative sentence capturing the claim
- `concepts`: list all concept IDs referenced by the observation
- Testable properties from notes.md are good candidates for observation claims

### Step 2.5: Extract Model Claims

For parameterized equation systems (multi-equation frameworks):

```yaml
- id: claim12
  type: model
  name: "LF model of glottal flow"
  equations:
    - "U_g(t) = E_0 * exp(alpha * t) * sin(omega_g * t)"
    - "U_g(t) = -E_e / (epsilon * T_a) * (exp(-epsilon * (t - T_e)) - exp(-epsilon * (T_c - T_e)))"
  parameters:
    - name: "E_e"
      concept: "concept3"
      note: "Excitation strength at maximum discontinuity"
    - name: "T_a"
      concept: "concept8"
      note: "Return phase time constant"
  provenance:
    paper: <paper_dir_name>
    page: <page number>
    section: "<section>"
```

### Step 2.6: Extract Measurement Claims

For perceptual or behavioral measurements:

```yaml
- id: claim15
  type: measurement
  target_concept: concept1
  measure: jnd_absolute
  value: 3.0
  unit: "Hz"
  listener_population: "native_english"
  methodology: "2AFC adaptive staircase"
  conditions:
    - "stimulus_type == 'synthetic_vowel'"
  provenance:
    paper: <paper_dir_name>
    page: <page number>
```

### Step 2.7: Assemble and Write

Assemble all claims into the ClaimFile structure:

```yaml
source:
  paper: <paper_dir_name>
  extraction_model: "<your model name>"
  extraction_date: "<today's date>"

claims:
  - id: claim1
    ...
```

Write to `<paper_dir>/claims.yaml`.

---

## Claim ID Rules

- Format: `claim` + sequential integer, no padding (claim1, claim2, ... claim103)
- IDs must be unique within a file
- When enriching, preserve existing IDs and continue the sequence for new claims
- Never reuse an ID, even if the original claim was removed

## Concept Naming Rules

- Check the concept registry first: `concepts/*.yaml` in the propstore
- Use the concept's `id` field (e.g., `concept1`), not its filename or canonical_name
- Check aliases: a paper might say "F0" which maps to `concept1` (fundamental_frequency)
- If no concept exists in the registry, use a descriptive `lowercase_underscore` name
- Do NOT create new concept registry entries — that is a separate workflow

## Provenance Rules

- `paper`: always the paper directory name (e.g., `Gobl_1988`)
- `page`: integer page number; use 0 only as last resort when page cannot be determined
- `section`: section name or number from the paper
- `table`: table identifier when claim comes from a table (e.g., "Table 2", "III")
- `figure`: figure identifier when claim comes from a figure
- `quote_fragment`: brief quote (1-2 sentences max) that directly supports the claim

## Quality Checklist

Before writing claims.yaml:
- [ ] Every claim has a unique sequential ID
- [ ] Every claim has type, provenance (with real page numbers where possible)
- [ ] Parameter claims have concept, value (or bounds), and unit
- [ ] Equation claims have expression, sympy (valid), and variable bindings
- [ ] Observation claims have statement and concepts list
- [ ] Model claims have name, equations list, and parameter bindings
- [ ] Measurement claims have target_concept, measure, value, and unit
- [ ] Conditions use consistent CEL vocabulary across the file
- [ ] Concept IDs match the registry where possible
- [ ] No speculative stances (only add what the paper supports)
- [ ] Notes added where methodological context matters

## Generating the Mechanical Baseline

If you need to generate the initial `claims.yaml` before enriching, run:

```bash
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/generate_claims.py <paper_dir>
```

This extracts parameters from tables, equations from `$$...$$` blocks, and observations from the Testable Properties section. The output has placeholder page numbers (0) and raw concept names — that is what enrich mode fixes.
